# 08. 커밋 메세지 관리

이 문서는 팀 프로젝트에서 Git 커밋 메세지를 일정한 형식으로 관리하기 위한 기준이다.

커밋 메세지를 통일하면 Git 기록을 읽기 쉽고, Jira 작업 코드와 연결해 어떤 작업에서 변경이 발생했는지 추적하기 쉽다.

---

## 1. 기본 형식

```text
타입: [Jira작업코드] 작업 내용 요약
```

예시:

```text
feat: [KAN-14] 지역별 전기차 등록 현황 조회 기능 추가
fix: [KAN-15] 충전소 데이터 CSV 인코딩 오류 수정
```

---

## 2. 커밋 타입 종류

| 타입 | 의미 | 사용 예시 |
|---|---|---|
| feat | 새로운 기능 추가 | 조회 기능, 검색 기능, 화면 기능 추가 |
| fix | 버그 수정 | 오류 수정, 잘못된 계산 수정, 예외 처리 수정 |
| refactor | 리팩토링 | 기능 변화 없이 코드 구조 개선 |
| docs | 문서 수정 | README, docs 문서 추가 또는 수정 |
| style | 코드 스타일 수정 | 들여쓰기, 공백, 포맷팅 수정 |
| test | 테스트 코드 추가/수정 | 테스트 파일 작성, 테스트 케이스 수정 |
| chore | 기타 작업 | 설정 파일 수정, 패키지 관리, 빌드 설정 변경 |
| build | 빌드 관련 수정 | Docker, 의존성, 빌드 스크립트 수정 |
| ci | CI/CD 설정 수정 | GitHub Actions, 배포 자동화 설정 수정 |
| db | 데이터베이스 관련 수정 | SQL, 마이그레이션, 테이블 구조 수정 |
| data | 데이터 파일 수정 | CSV, JSON, Excel 원본 데이터 수정 |
| design | UI/UX 수정 | 화면 배치, 색상, 디자인 요소 수정 |
| perf | 성능 개선 | 속도 개선, 쿼리 최적화, 불필요한 반복 제거 |
| revert | 이전 커밋 되돌리기 | 잘못된 커밋 롤백 |

---

## 3. Jira 작업 코드 작성 기준

Jira 작업 코드가 있는 경우 커밋 메세지에 반드시 포함한다.

```text
feat: [KAN-14] 지역별 전기차 등록 현황 조회 기능 추가
```

여기서 `KAN-14`는 Jira의 작업 코드이다.

- `KAN`: Jira 프로젝트 키
- `14`: 작업 번호

Jira 작업 코드가 있으면 다음 내용을 추적하기 쉽다.

- 어떤 기능 작업에서 변경되었는지
- 어떤 버그 수정과 연결되는지
- 팀원이 어떤 Jira 이슈를 보고 작업했는지
- 나중에 Git 기록과 Jira 작업 내역을 함께 확인할 수 있는지

---

## 4. 대표 커밋 예시

### feat

새로운 기능을 추가할 때 사용한다.

```text
feat: [KAN-14] 지역별 전기차 등록 현황 조회 기능 추가
feat: [KAN-18] 충전소 현황 검색 기능 추가
feat: [KAN-22] 차종별 전기차 정보 입력 기능 추가
```

### fix

버그나 잘못된 동작을 수정할 때 사용한다.

```text
fix: [KAN-15] CSV 파일 한글 깨짐 오류 수정
fix: [KAN-19] 지역 선택 시 빈 데이터가 출력되는 문제 수정
fix: [KAN-23] 등록 대수 증가율 계산 오류 수정
```

### refactor

기능 동작은 그대로 두고 코드 구조만 개선할 때 사용한다.

```text
refactor: [KAN-16] 데이터 로딩 로직 함수 분리
refactor: [KAN-20] Streamlit 화면 출력 코드 정리
refactor: [KAN-24] 중복된 CSV 처리 로직 통합
```

### docs

문서만 추가하거나 수정할 때 사용한다.

```text
docs: [KAN-17] 프로젝트 README 요구사항 설명 추가
docs: [KAN-21] 크롤링 문서 작성 방식 정리
docs: [KAN-25] 커밋 메세지 관리 문서 추가
```

### style

코드 동작에 영향이 없는 스타일 수정에 사용한다.

```text
style: [KAN-26] Python 코드 들여쓰기 정리
style: [KAN-27] 불필요한 공백 제거
style: [KAN-28] 변수 선언 위치 정리
```

### test

테스트 코드와 관련된 작업에 사용한다.

```text
test: [KAN-29] 데이터 변환 함수 테스트 추가
test: [KAN-30] CSV 로딩 예외 테스트 추가
```

### chore

기능, 버그, 문서가 아닌 기타 관리 작업에 사용한다.

```text
chore: [KAN-31] requirements.txt 패키지 목록 정리
chore: [KAN-32] .gitignore 설정 추가
chore: [KAN-33] 프로젝트 폴더 구조 정리
```

### db

데이터베이스 구조나 SQL 변경에 사용한다.

```text
db: [KAN-34] 전기차 등록 현황 테이블 생성
db: [KAN-35] 충전소 정보 테이블 컬럼 추가
db: [KAN-36] Flyway 마이그레이션 파일 추가
```

### data

데이터 파일 자체를 추가하거나 수정할 때 사용한다.

```text
data: [KAN-37] 전기차 등록 현황 CSV 파일 추가
data: [KAN-38] 충전소 위치 데이터 컬럼명 수정
data: [KAN-39] 분석용 샘플 데이터 갱신
```

## 5. 커밋 메세지 작성 규칙

커밋 메세지는 짧고 명확하게 작성한다.

좋은 예시:

```text
feat: [KAN-14] 지역별 전기차 등록 현황 조회 기능 추가
```

나쁜 예시:

```text
수정함
작업함
파일 변경
feat 작업
```

작성 기준:

- 한 커밋에는 하나의 목적만 담는다.
- 커밋 타입을 먼저 작성한다.
- Jira 작업 코드가 있으면 `[KAN-14]` 형식으로 작성한다.
- 작업 내용을 너무 길게 쓰지 않는다.
- 무엇을 변경했는지 알 수 있게 작성한다.

---

## 6. 우리 프로젝트 권장 형식

우리 프로젝트에서는 아래 형식을 기본으로 사용한다.

```text
타입: [KAN-번호] 작업 내용
```

예시:

```text
feat: [KAN-14] 지역별 전기차 등록 현황 조회 기능 추가
fix: [KAN-15] 충전소 데이터 로딩 오류 수정
refactor: [KAN-16] 데이터 전처리 함수 분리
docs: [KAN-17] 커밋 메세지 관리 문서 추가
db: [KAN-18] 전기차 등록 현황 테이블 생성
data: [KAN-19] 분석용 CSV 데이터 추가
```

Jira 작업 코드가 아직 없는 작업이라면 임시로 아래처럼 작성할 수 있다.

```text
docs: 커밋 메세지 관리 문서 초안 작성
```

하지만 팀 작업에서는 가능하면 Jira 작업 코드를 만든 뒤 커밋하는 것을 권장한다.
